# Art Tool 首次版本更新指导（V1 Update Guide）

本文档用于指导 Art Tool 在现有 MVP 基础上完成第一次功能更新，目标是新增两项核心能力：

1. 批量重命名：支持自定义命名和模板命名。
2. 用户自定义工作流：支持将多个图片处理任务按顺序编排并执行。

## 1. 更新目标

- 在不破坏现有单功能流程的前提下，扩展为“快捷模式 + 工作流模式”。
- 让用户可以只做重命名，也可以把重命名作为工作流中的一个步骤。
- 在任务进度、结果报告中提供步骤级可观测信息（正在执行哪一步、哪一步失败）。

## 2. 范围定义

### 2.1 本次包含

- 批量重命名规则：
  - 自定义命名（固定前后缀、序号、扩展名保留）。
  - 模板命名（花括号占位符）。
- 工作流顺序执行：
  - 每个文件按步骤串行执行（step1 -> step2 -> step3）。
  - 支持步骤增删和顺序调整（首版建议上移/下移按钮）。
- 兼容旧流程：
  - 保留现有“单功能一键处理”入口。
  - 旧请求结构仍可运行。

### 2.2 本次不包含

- 条件分支工作流（if/else）。
- 并行分支工作流。
- 循环节点。
- 云端工作流与跨设备同步。

## 3. 已确认技术决策

- 框架保持不变：Tauri + Rust + React/TypeScript。
- 模板语法采用花括号：`{name}`、`{index}`、`{date}`、`{time}`、`{ext}`。
- 工作流执行模型：按文件串行执行步骤。
- 兼容策略：保留现有单功能快捷模式。

## 4. 设计总览

### 4.1 后端执行模型

从当前“单处理器一次执行”升级为“工作流步骤链执行”：

- 当 `workflowSteps` 为空时：走旧逻辑（单处理器）。
- 当 `workflowSteps` 非空时：
  - 对每个输入文件逐步执行每个处理器。
  - 维护中间输出路径。
  - 最终输出由命名规则引擎统一生成。

### 4.2 命名规则引擎

在输出路径规划中引入统一命名规则：

- 输入：原文件信息 + 重命名规则 + 批处理索引。
- 输出：最终文件名（含冲突消解）。
- 规则：
  - Windows 非法字符清洗：`\\ / : * ? \" < > |`。
  - 文件名超长保护（必要时截断并保留扩展名）。
  - 冲突策略：自动追加 `_1`、`_2`、`_3`。

### 4.3 前端交互模型

前端由“单处理器选择”扩展为两种模式：

- 快捷模式：保持现状，适合简单任务。
- 工作流模式：
  - 可添加多个处理步骤。
  - 每步独立参数。
  - 可在参数区配置重命名规则。

## 5. 分阶段执行计划

### 阶段 A：协议与数据结构改造（先做）

目标：先打通数据模型和兼容逻辑。

- 扩展请求类型：新增 `workflowSteps`、`renameConfig`（可选）。
- 扩展报告类型：支持步骤级字段。
- 兼容逻辑：旧字段仍可触发单处理器运行。

建议改动文件：

- `src/lib/types.ts`
- `src-tauri/src/commands/batch.rs`
- `src-tauri/src/core/report.rs`

### 阶段 B：命名引擎与重命名能力

目标：可独立完成批量重命名。

- 抽离命名规划模块（模板解析 + 冲突消解）。
- 实现重命名处理器或同等步骤执行器。
- 接入现有输出路径规划。

建议改动文件：

- `src-tauri/src/core/output_resolver.rs`
- `src-tauri/src/core/processors/mod.rs`
- `src-tauri/src/core/processors/rename.rs`（新增）
- `src-tauri/src/core/registry.rs`

### 阶段 C：工作流执行引擎

目标：完成多步骤顺序执行。

- 在批处理执行器内引入步骤循环。
- 单文件失败隔离保持不变。
- 增加步骤级进度回传。

建议改动文件：

- `src-tauri/src/core/batch_job_runner.rs`
- `src-tauri/src/core/processor.rs`（必要时扩展上下文字段）
- `src-tauri/src/core/file_discovery.rs`（按步骤能力调整过滤）

### 阶段 D：前端工作流 UI 与状态管理

目标：用户可在 UI 中配置并运行工作流。

- 新增工作流构建组件。
- 新增重命名配置组件。
- Store 改为按步骤实例存参数，避免同处理器参数覆盖。

建议改动文件：

- `src/store/taskStore.ts`
- `src/App.tsx`
- `src/lib/api/tauri.ts`
- `src/components/WorkflowBuilder.tsx`（新增）
- `src/components/RenameRulePanel.tsx`（新增）
- `src/components/TaskQueuePanel.tsx`
- `src/components/ResultSummaryPanel.tsx`

### 阶段 E：测试与文档收口

目标：保证可回归、可交付。

- 新增命名模板测试。
- 新增工作流顺序执行测试。
- 新增步骤失败隔离测试。
- 更新用户文档和接口文档。

建议改动文件：

- `src-tauri/tests/rename_tests.rs`（新增）
- `src-tauri/tests/workflow_runner_tests.rs`（新增）
- `src-tauri/tests/batch_job_runner_tests.rs`

## 6. Copilot 可执行任务清单

1. 扩展前后端请求/响应类型，加入 `workflowSteps` 和 `renameConfig`。
2. 在后端实现模板解析器与文件名冲突处理。
3. 新增 `rename` 处理能力并注册到处理器注册器。
4. 重构批处理执行器，支持单文件内多步骤顺序执行。
5. 扩展进度事件和报告结构，增加步骤级信息。
6. 新增工作流构建 UI（步骤增删、排序、参数编辑）。
7. 新增重命名规则 UI（自定义名、模板名、预览）。
8. 保留并验证旧快捷模式。
9. 补齐 Rust 测试与手工验证清单。
10. 更新 README 与接口文档。

## 7. 验收标准

- 可对至少 100 张图片执行工作流任务，流程稳定。
- 重命名支持自定义与模板，输出命名正确且冲突可自动消解。
- 单文件任一步骤失败不影响其他文件完成。
- 任务队列可显示步骤级进度信息。
- 旧单功能快捷模式可继续正常运行。
- 在 Windows 干净环境可正常安装并运行。

## 8. 风险与缓解

- 中间文件过多导致磁盘压力：
  - 采用临时目录并在单文件完成后清理。
- 工作流参数复杂导致易错：
  - 每步独立参数校验，前端提供默认值和实时提示。
- 命名规则引发路径异常：
  - 统一非法字符清洗、长度保护、重名追加序号。
- 兼容改造引起回归：
  - 强制保留旧模式回归测试用例。

## 9. 推荐执行顺序

1. 先做协议兼容和类型改造（阶段 A）。
2. 再做命名引擎（阶段 B）。
3. 再做工作流执行器（阶段 C）。
4. 然后做前端 UI（阶段 D）。
5. 最后测试和文档收口（阶段 E）。

这样可以保证每一步都有可验证输出，降低一次性改动风险。
